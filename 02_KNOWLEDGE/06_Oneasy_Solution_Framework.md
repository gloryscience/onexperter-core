# Oneasy Solution Framework

## 1. Purpose

Provide controlled, current and scenario-based mapping from customer problems to Oneasy.

## 2. Positioning

Oneasy is a SaaS platform for multi-vendor optical access operations, focused on small and mid-sized ISPs. It is not intended to replace every vendor NMS or become a large customized Tier-1 OSS by default.

## 3. Value Areas

- multi-vendor OLT and ONU/CPE management;
- standardized and batch operations;
- remote diagnosis and customer support;
- TR-069/TR-369-based CPE management where supported;
- SaaS tenant, user, entitlement, subscription and audit capabilities;
- AI-assisted O&M where implemented and validated.

## 4. Mapping Method

```text
Customer problem → required capability → data/device prerequisites
→ current Oneasy support → gaps → trial/POC validation → decision
```

## 5. Do Not Overclaim

Do not claim support for every vendor/model/firmware, all faults, unrestricted automation, replacement of engineers/vendor NMS, guaranteed compatibility or unimplemented digital-twin functions.

## 6. Trial Logic

A trial should use real equipment, a named owner, success criteria, timeline, data access and conversion decision. It is not indefinite free production use.

## 7. Digital Twin Direction

OnExpert provides expert/twin intelligence; Oneasy may provide resource data, telemetry, workflows and controlled tools. Current implementation must be checked against release documentation.

## 8. GPT Behavior

Provide independent analysis first. State when Oneasy is relevant, prerequisites, limitations and alternatives. Say clearly when it is not a fit.
