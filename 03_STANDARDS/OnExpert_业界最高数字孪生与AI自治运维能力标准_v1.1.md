# OnExpert 业界最高数字孪生与 AI 自治运维能力标准

> **标准名称**：OnExpert Digital Twin & AI-native Autonomous OAM Capability Standard  
> **简称**：OE-DTAO  
> **版本**：v1.1  
> **文档类型**：核心能力标准 / 产品与技术顶层设计 / 研发验收基线  
> **适用范围**：OLT、ONU/ONT、CPE、ODN、PON 接入网、接入业务及相关 OSS/OAM 场景  
> **标杆对象**：Huawei、Nokia、Ericsson  
> **标准定位**：吸收国际头部通信厂商的网络数字孪生、自治网络和工程数字化能力，形成面向多厂商光接入网、AI 原生、厂商中立、可解释、可验证、可执行的 OnExpert 最高能力标准。  
> **说明**：本标准为 OnExpert 内部能力标准，不代表 ITU、TM Forum 或任何设备厂商的官方认证标准。

---

## 一、核心定位

OnExpert 的目标不是创建更精细的三维设备展示，而是建立：

> **面向多厂商光接入网络，从机房、机架、机框、板卡、端口、ODN、ONU到客户业务贯通的工程级数字孪生与 AI 自治运维体系。**

最高能力必须同时具备：

```text
真实映射
+ 持续同步
+ 因果理解
+ 仿真预测
+ 意图驱动
+ 安全执行
+ 结果验证
+ 持续学习
```

完整闭环为：

```text
感知 → 建模 → 理解 → 推演 → 决策 → 执行 → 验证 → 学习
```

OnExpert 的差异化不应停留在“会管理更多厂商设备”，而应体现为：

> **以工程级数字孪生作为网络事实底座，以 AI Agent 作为认知与执行引擎，以仿真和安全护栏约束生产网络操作。**

## 三、OnExpert 的原创定位

OnExpert 不应成为大型厂商数字孪生产品的缩小版，而应形成以下原创定位：

### 3.1 面向光接入网的细粒度孪生

大型厂商通常侧重网络域和控制器层，OnExpert 应进一步下沉到：

```text
机房 → 机架 → 机框 → 槽位 → 板卡 → 模块 → 端口 → 光模块
→ ODF → 光缆 → 分光器 → ONU → 家庭网关 → 客户业务
```

以设备级、光链路级、客户级和业务级对象贯通形成差异化。

### 3.2 多厂商语义统一

不同设备厂商在 MIB、CLI、告警码、性能指标、端口命名、ONU 状态和配置模型方面存在明显差异。OnExpert 应建立厂商中立的统一语义层，使不同厂商数据映射到统一的：

```text
Resource / State / Alarm / Metric / Configuration / Topology / Service / Action
```

模型。

### 3.3 中小 ISP 可消费的数字孪生

OnExpert 的数字孪生不应依赖大型集成项目和长期定制，应采用 SaaS 化、模板化、自动发现、渐进建模、按需加载、低成本接入、标准 API 和可配置工作流，使中小 ISP 也能使用高等级数字孪生和自治运维能力。

## 四、AI 与数字孪生融合的原创能力体系

OnExpert 应形成以下十项 AI 原生创新能力。

### 4.1 TwinGraph：因果孪生图谱

传统数字孪生主要表达“对象是什么、连接在哪里”。OnExpert 应进一步建立因果图谱：

```text
设备 → 端口 → 光链路 → ONU → 配置 → 业务 → 客户体验 → 告警 → 故障原因
```

TwinGraph 不只是拓扑图，而是表达：

- 依赖关系；
- 承载关系；
- 配置传播关系；
- 故障传播关系；
- 性能影响关系；
- 时间先后关系；
- 因果置信度。

AI Agent 的诊断、预测和决策必须以 TwinGraph 为事实基础，而不是只基于自然语言或日志摘要。

### 4.2 TwinCopilot：面向工程对象的对话式运维

工程师可以直接询问：

- 哪个 PON 口正在恶化；
- 为什么这批 ONU 同时离线；
- 某板卡更换会影响哪些客户；
- 上联拥塞是否会影响 IPTV；
- 当前最值得优先处理的风险是什么。

系统回答必须自动关联对象、拓扑、指标、告警、历史、配置、业务和工单。输出不能只有自然语言，还应同步高亮相关设备、路径和受影响对象。

### 4.3 Self-Calibrating Twin：自校准孪生

数字孪生模型会因设备变更、板卡更换、光纤调整和配置漂移而失真。

OnExpert 应通过 AI 自动发现：

- 资产清单与实际设备不一致；
- 端口连接关系发生变化；
- ONU 归属变化；
- ODF 与分光器数据冲突；
- 配置模型与设备实际状态不一致；
- 拓扑长期未更新。

系统应生成：

```text
模型差异 → 证据 → 建议修正 → 置信度 → 人工确认 → 模型更新
```

实现数字孪生的持续校准。

### 4.4 Counterfactual Twin：反事实推演

系统不仅回答“发生了什么”，还应回答：

- 如果关闭该 PON 口会怎样；
- 如果更换板卡会影响多少 ONU；
- 如果分光比从 1:64 调整为 1:32，光功率和容量会如何变化；
- 如果新增 XGS-PON 板卡，上联是否成为瓶颈；
- 如果执行某条配置，哪些业务可能中断。

反事实推演应输出假设条件、影响对象、预期变化、风险等级、不确定性和是否允许执行。

### 4.5 SafeAction Sandbox：安全动作沙箱

任何 AI Agent 生成的配置或操作，必须先在数字孪生沙箱中验证。

标准流程：

```text
意图 → AI 生成动作计划 → 数字孪生仿真 → 风险检查 → 影响分析
→ 审批 → 小范围执行 → 结果验证 → 扩大执行或回滚
```

沙箱必须检查：

- 配置语法；
- 设备兼容性；
- 前置条件；
- 业务影响；
- 容量冲突；
- 权限；
- 维护窗口；
- 回滚可行性。

### 4.6 Uncertainty-aware AI：带不确定性的 AI

OnExpert 不应输出绝对化结论。每个诊断和预测必须区分：

- 已确认事实；
- 高可信推断；
- 低可信假设；
- 缺失数据；
- 冲突数据；
- 无法判断项。

标准输出：

```text
结论 / 证据 / 置信度 / 替代解释 / 缺失信息 / 建议验证步骤
```

### 4.7 Temporal Twin：时间孪生

数字孪生不能只保存当前状态，还应保存网络状态随时间的演进。

Temporal Twin 应支持：

- 故障发生前状态；
- 故障发生时状态；
- 故障恢复后状态；
- 配置变更前后对比；
- 光功率缓慢劣化；
- 温度与流量相关性；
- ONU 掉线周期；
- 告警传播时间线。

AI 诊断应优先分析事件顺序和变化趋势，避免只看单一时点。

### 4.8 Multimodal Twin Builder：多模态自动建模

OnExpert 应利用官方手册、PDF 图纸、设备照片、标签照片、CLI 输出、MIB 文件、资产清单和现场测量，自动辅助生成设备模板、板卡库、端口模型和资源关系。

自动生成结果必须经过：

```text
证据引用 → 规则校验 → 工程师确认 → 发布
```

不得直接把视觉识别或大模型推断当作正式设备事实。

### 4.9 Collective Learning：跨网络知识学习

在不泄露客户数据和商业秘密的前提下，OnExpert 应沉淀：

- 厂商型号兼容经验；
- 典型告警模式；
- 光功率劣化模式；
- ONU 掉线模式；
- 配置错误模式；
- 处理方案和验证结果。

形成：

```text
单客户经验 → 匿名化模式 → 通用诊断知识 → 其他客户复用
```

该能力必须遵守租户隔离、匿名化、授权和数据治理要求。

### 4.10 Twin-to-Knowledge：孪生到知识资产

每次故障、变更和处理完成后，系统应自动生成：

- 事件摘要；
- 根因；
- 证据链；
- 处理步骤；
- 结果验证；
- 适用设备和版本；
- 风险边界；
- 可复用知识条目。

数字孪生不仅记录状态，也应持续转化为 OnExpert 的专业知识资产。

## 五、最高能力架构

OnExpert 最高标准采用九层架构。

```text
L9 业务意图与自治运营层
L8 AI Agent与决策编排层
L7 仿真、预测与安全验证层
L6 保障、诊断与影响分析层
L5 网络与业务数字孪生层
L4 因果图谱与时间孪生层
L3 统一数据与知识模型层
L2 多厂商接入与控制层
L1 物理网络与工程资产层
```

### 5.1 L1：物理网络与工程资产层

覆盖 Site、Room、Rack、Chassis、Slot、Board、Module、Port、ODF、Cable、Splitter、ONU/ONT/CPE、Power、Fan 和 Environment Sensor。

要求：稳定唯一 ID、工程尺寸和结构、版本和生命周期、证据来源、真实性等级。

### 5.2 L2：多厂商接入与控制层

支持 SNMP、CLI、NETCONF/YANG、REST API、Syslog、TR-069/TR-369、厂家 NMS/EMS、文件和批量导入。

必须具备厂商适配器、协议转换、语义标准化、命令安全代理、凭据保护和采集质量监控。

### 5.3 L3：统一数据与知识模型层

统一表达：

```text
Resource / Topology / State / Alarm / Metric / Configuration
Service / Customer / Action / Event / Knowledge
```

要求模型可扩展、厂商中立、版本可追踪、数据血缘可追踪，实时、历史、模拟和推断数据可区分。

### 5.4 L4：因果图谱与时间孪生层

要求具备对象依赖关系、故障传播关系、业务承载关系、配置影响关系、时间序列、事件时间线、因果置信度和模型自校准。

### 5.5 L5：网络与业务数字孪生层

必须贯通：

```text
设备 → 端口 → ODN → ONU → VLAN → 上联 → BNG/BRAS → 客户业务
```

支持设备视图、物理拓扑、逻辑拓扑、业务路径、客户影响、容量与健康度。

### 5.6 L6：保障、诊断与影响分析层

提供告警关联、异常检测、根因分析、影响分析、服务质量分析、故障定位、预测性维护和处理建议。

### 5.7 L7：仿真、预测与安全验证层

提供配置 Dry Run、容量仿真、故障传播仿真、维护影响仿真、反事实推演、回滚验证和动作沙箱。

### 5.8 L8：AI Agent与决策编排层

Agent 应按角色分工：

- Asset Agent；
- Alarm Agent；
- Diagnosis Agent；
- Optical Agent；
- Capacity Agent；
- Configuration Agent；
- Security Agent；
- Change Agent；
- Customer Experience Agent；
- Knowledge Agent。

Agent 之间通过统一孪生上下文协作，不能各自直接读取碎片化数据并独立作结论。

### 5.9 L9：业务意图与自治运营层

支持业务意图：提升 ONU 在线率、降低频繁掉线、降低 PON 端口拥塞、缩短 MTTR、降低上门维护、提升客户体验、控制能耗和降低变更风险。

系统应将业务意图分解为：

```text
目标 → KPI → 网络对象 → 诊断 → 动作计划 → 仿真 → 审批 → 执行 → 验证
```

### 6.1 数字孪生成熟度 DT0–DT5

| 等级 | 能力 |
|---|---|
| DT0 | 静态资料与二维展示 |
| DT1 | 外观和资产数字化 |

| DT4 | 因果分析、预测与仿真 |
| DT5 | 自校准、跨域、业务级高保真孪生 |

### 6.2 自治运维成熟度 AO0–AO5

| 等级 | 能力 |
|---|---|
| AO0 | 人工操作 |
| AO1 | 工具辅助 |
| AO2 | 局部自动化 |
| AO3 | 条件闭环与人工审批 |

| 阶段 | 目标 |

## 七、AI 输出和动作标准

### 7.1 诊断输出

必须包括：

```text
症状 / 异常对象 / 时间 / 证据 / 可能原因 / 置信度
替代解释 / 影响范围 / 建议动作 / 验证方法 / 风险边界
```

### 7.2 动作计划

必须包括：

- 动作对象；
- 动作内容；
- 前置条件；
- 影响范围；
- 风险等级；
- 仿真结果；
- 审批要求；
- 回滚方案；
- 执行窗口；
- 验证指标。

### 7.3 禁止直接自治的操作

未经明确授权和审批，不得自动执行：

- 板卡复位；
- PON 端口关闭；
- 批量 ONU 配置；
- VLAN 删除；
- 固件升级；
- 主备切换；
- 批量重启；
- 路由和上联重大变更；
- 生产网络脚本。

---

## 八、开放性标准

OnExpert 必须提供标准 REST API、Webhook、事件流、模型查询接口、拓扑查询接口、诊断 API、仿真 API、Agent 工具接口、数据导入导出和第三方应用扩展机制。

模型和接口应优先参考 YANG、OpenConfig、TM Forum Open API、Broadband Forum 数据模型、厂家 MIB 和北向接口。

不得把 OnExpert 构建为另一套新的数据孤岛。

---

## 九、安全与可信标准

必须实现：

- 租户隔离；
- 最小权限；
- 只读与执行权限分离；
- 凭据加密；
- 命令代理；
- 操作审批；
- 双人复核；
- 审计日志；
- 数据血缘；
- 模型版本管理；
- AI 提示和工具调用审计；
- 模拟、推断和真实数据标识；
- 高风险动作强制仿真；
- 自动回滚或安全停止。

AI 系统必须防范模型幻觉、提示注入、错误工具调用、过期拓扑、错误设备映射、权限越界、恶意配置和不可信第三方数据。

## 十、最高标准验收要求

达到 OnExpert 最高标准时，系统应能完成以下任务：

1. 从机房定位到设备、板卡和端口；
2. 从客户业务反查完整物理与逻辑路径；
3. 实时显示设备、ODN、ONU和业务状态；
4. 识别模型与现网差异并提出校准建议；
5. 基于时间线和因果图谱诊断故障；
6. 计算故障对 ONU、客户和业务的影响；
7. 对配置、维护和扩容方案进行反事实推演；
8. 在执行前完成风险、兼容性和回滚验证；
9. 通过 AI Agent 生成可解释动作计划；
10. 在授权范围内执行并自动验证结果；
11. 将处理结果沉淀为可复用知识；
12. 支持多厂商设备和跨域系统；
13. 具备明确的不确定性、数据来源和证据链；
14. 所有关键动作可审计、可撤销、可复盘。

## 十一、OnExpert 的核心特色表达

> **Twin-grounded AI**：所有 AI 判断建立在数字孪生事实底座之上。

> **AI-calibrated Twin**：AI 持续发现和修正数字孪生与真实网络之间的偏差。

> **Simulation-guarded Action**：所有高风险动作先仿真、再审批、后执行、再验证。

> **Causal and Explainable OAM**：诊断不仅给出结论，还给出因果链、证据和不确定性。

> **Vendor-neutral Access Intelligence**：以多厂商统一语义构建厂商中立的光接入网智能。

---

## 十二、最终标准

OnExpert 业界最高能力标准可以浓缩为：

```text
看得见 → 对得上 → 看得懂 → 推得准 → 做得稳 → 验得明 → 学得会
```

其中：

- **看得见**：真实、完整地数字化网络；
- **对得上**：数字模型与现网持续一致；
