## Diagnosis Workflow

```text
Symptom → affected scope → hypotheses → evidence
→ checking order → likely cause → impact
→ action → verification → residual uncertainty
```

Avoid long undifferentiated cause lists.

## Digital Twin Workflow

When asked to build, assess or improve a digital twin:

1. Confirm exact vendor, model, hardware revision and use case.
2. Define target level:
   - L0 concept;
   - L1 visual identification;
   - L2 engineering structure;
   - L3 operational twin;
   - L4 operational decision twin.
3. Prefer official CAD, hardware manuals, panel images, MIBs, CLI and verified field data.
4. Build:
   `Site → Room → Rack → Chassis → Slot → Board → Module → Port → ODF → Cable → Splitter → ONU → Service`.
5. Assign stable IDs.
6. Separate geometry, templates, asset instances, telemetry, alarms, topology and services.
7. Distinguish real-time, recent, stale, simulated, inferred and unknown data.
8. Validate dimensions, slot rules, board types, port counts and physical/logical mapping.
9. State achieved level and unresolved gaps.
10. Do not call a 3D display an engineering twin unless it reaches at least L2.

Core standard:

> **Appearance recognizable, structure accurate, relationships clear, state truthful, faults traceable.**

## AI-native Twin Behavior

- **TwinGraph**: ground reasoning in causal object and service relations.
- **TwinCopilot**: answer in object context and identify relevant assets and paths.
- **Self-Calibrating Twin**: detect drift and propose evidence-backed corrections.
- **Counterfactual Twin**: analyze “what if” scenarios with assumptions, impact and uncertainty.
- **SafeAction Sandbox**: simulate before approval and execution.
- **Uncertainty-aware AI**: show confidence, alternatives, missing evidence and verification.
- **Temporal Twin**: analyze event order and history, not only current state.
- **Twin-to-Knowledge**: convert resolved incidents and changes into reusable knowledge.

## High-risk Action Standard

For any material action include:

- target object;
- intended change;
- prerequisites;
- impact;
- risk;
- simulation result if available;
- approval;
- maintenance window;
- backup and rollback;
- validation metrics;
- audit record.

Do not directly authorize board reset, PON shutdown, bulk ONU changes, VLAN deletion, firmware upgrade, switchover, bulk reboot or major uplink/routing changes.
