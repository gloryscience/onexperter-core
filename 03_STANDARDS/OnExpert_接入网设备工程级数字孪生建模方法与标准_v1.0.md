> **文档类型**：核心知识资产 / 工程方法论 / 建模与验收标准

## 1. 目的

本文件用于规范接入网设备工程级数字孪生的建模方法、实施流程、数据结构、交互方式、质量要求与验收标准。

工程级数字孪生不是设备三维展示，也不是简单的资产看板。其目标是把设备的：

- 物理结构；
- 逻辑资源；
- 连接关系；
- 运行状态；
- 告警与性能；
- 业务影响；
- 运维动作；

统一映射为可查询、可分析、可追踪、可诊断的数字对象。

最终应使用户或工程师能够在较短时间内回答以下问题：

1. 设备在哪里；
2. 设备由哪些板卡、模块和端口组成；
3. 当前安装配置与标准配置是否一致；
4. 哪个对象正在异常；
5. 异常影响哪些端口、ONU、客户和业务；
6. 设备当前容量、温度、功耗、带宽和健康度如何；
7. 下一步应检查、处理或升级什么。

## 2. 工程级数字孪生的定义

接入网设备工程级数字孪生，是对真实设备及其运行环境建立的多层数字映射，至少包括四个同步维度：

```text
物理实体
+ 逻辑资源
+ 运行状态
+ 业务关系
```

其基本结构为：

```text
真实设备
→ 标准化资源模型
→ 几何与面板模型
→ 实时遥测与告警
→ 业务拓扑与影响关系
→ 运维诊断与操作
```

工程级数字孪生必须同时具备：

| 维度 | 基本要求 |
|---|---|
| 几何真实性 | 外形、尺寸、比例、槽位、端口位置能够对应真实设备 |
| 资源完整性 | 机架、机框、槽位、板卡、模块、端口均为独立对象 |
| 逻辑一致性 | 物理端口与逻辑接口、PON、VLAN、业务路径可关联 |
| 状态动态性 | 告警、性能、温度、功耗、在线率等可更新 |
| 影响可追踪 | 故障可关联到下游 ONU、客户和业务 |
| 运维可执行 | 可提供诊断、检查、维护、变更和回滚入口 |

## 3. 建模最高准则

### 3.1 准确性优先于视觉效果

设备外观的视觉质量不能代替工程准确性。任何无法确认的信息不得伪装成真实结构。

处理原则：

- 官方资料明确的，按事实建模；
- 实物照片可确认的，按照片校准；
- 行业规律可合理推断的，标记为“工程推断”；
- 无法确认的，采用占位模型并明确标记；
- 不得根据设备名称自行编造板卡、端口或内部结构。

### 3.2 物理模型与业务数据分离

几何模型、资源模板、实际资产、运行数据必须分离。

推荐结构：

```text
GLB / glTF
→ 外观、结构、空间和材质

Device Template JSON
→ 槽位规则、板卡规则、端口规则、兼容规则

Asset Instance
→ 实际设备、序列号、槽位配置、软件版本

Telemetry / Alarm
→ 实时性能、告警、温度、功耗、光功率

Topology / Service Graph
→ ODF、光缆、分光器、ONU、VLAN、业务关系
```

不得把实时状态、板卡型号和端口属性直接写死在三维模型中。

### 3.3 对象必须可独立识别

每个重要对象必须具有稳定、唯一、可持久化的资源 ID。

推荐格式：

```text
site/{site-id}/room/{room-id}/rack/{rack-id}/chassis/{device-id}
```

继续向下扩展：

```text
.../slot/{slot-id}/board/{board-id}/port/{port-id}
```

PON 端口示例：

```text
site/wh01/room/a/rack/r03/chassis/olt001/slot/01/board/gnoa01/pon/01
```

资源 ID 不应随界面名称、语言或显示顺序变化。

## 4. 标准对象分层

### 4.1 空间与机房层

```text
Site
└─ Building
   └─ Room
      └─ Row
         └─ Rack
```

应记录：

- 站点名称和位置；
- 机房、列、机架编号；
- 机架规格和 U 位；
- 供电、接地、空调、环境传感器；
- 走线架和 ODF 相对位置。

### 4.2 设备物理层

```text
Rack
└─ Chassis
   ├─ Slot
   │  └─ Board
   │     ├─ Module
   │     ├─ Physical Port
   │     └─ LED / Indicator
   ├─ Power Module
   ├─ Fan Unit
   ├─ Control Board
   ├─ Backplane
   └─ Grounding Point
```

### 4.3 接入网络层

```text
Physical Port
└─ PON Interface
   └─ ODF Port
      └─ Fiber Cable
         └─ Splitter
            └─ Drop Cable
               └─ ONU / ONT
```

### 4.4 业务逻辑层

```text
ONU
→ T-CONT
→ GEM Port
→ Service Port
→ VLAN / QinQ
→ Uplink Interface
→ Aggregation
→ BNG / BRAS / DHCP / Radius
```

### 4.5 运维对象层

```text
Asset
├─ Alarm
├─ Performance Metric
├─ Event
├─ Configuration
├─ Software Version
├─ Work Order
├─ Change
├─ Diagnostic Result
└─ Maintenance Record
```

## 5. 模型范围与精度等级

接入网设备数字孪生应根据用途确定精度，不宜一开始全部追求最高精度。

| 等级 | 定位 | 适用场景 |
|---|---|---|
| L0 | 概念模型 | 产品说明、方案沟通 |
| L1 | 外观识别模型 | 培训、资产展示、设备识别 |
| L2 | 工程结构模型 | 机架、槽位、板卡、模块、端口管理 |
| L3 | 运行数字孪生 | 告警、性能、温度、功耗、容量、历史状态 |
| L4 | 运维决策孪生 | 故障诊断、影响分析、变更、回滚、工单联动 |

OnExpert 默认所称“工程级数字孪生”，最低应达到 **L2**；正式运行孪生应达到 **L3**；具备运维价值的目标版本应达到 **L4**。

## 6. 标准建模流程

## 6.1 第一步：明确用途和边界

先定义模型是用于：

- 资产展示；
- 培训；
- 设备认知；
- 机房管理；
- 实时监控；
- 故障诊断；
- 维护操作；
- 容量规划；
- Oneasy 产品集成。

同时明确：

- 是否接入真实设备；
- 是否需要端口级；
- 是否需要 ONU 和业务映射；
- 是否需要历史回放；
- 是否允许执行操作。

没有明确用途，不应直接开始三维建模。

## 6.2 第二步：收集证据

资料优先级：

| 等级 | 资料类型 |
|---|---|
| A | 厂家 CAD、机械图、官方硬件说明书、正式数据手册 |
| B | 官方高清照片、安装手册、板卡手册、MIB 手册 |
| C | 实物测量、工程现场照片、设备标签、CLI 输出 |
| D | 经多个来源验证的公开图片或渠道资料 |
| E | 工程推断，仅用于占位 |

必须建立《模型证据表》，记录：

- 来源；
- 日期；
- 适用型号；
- 可信等级；
- 已确认内容；
- 未确认内容。

## 6.3 第三步：建立设备模板

设备模板至少包括：

- 设备型号；
- 机框尺寸；
- 安装方式；
- 槽位编号；
- 支持板卡类型；
- 电源模块；
- 风扇结构；
- 公共接口；
- 端口模板；
- 最大容量；
- 功耗和温度边界；
- 支持的软件和协议。

设备模板描述“这种设备可以是什么”，不描述某一台真实设备当前是什么。

## 6.4 第四步：建立板卡与模块库

每种板卡必须独立建模，至少记录：

```text
型号
物料号
板卡类型
可安装槽位
面板尺寸
端口数量
端口类型
连接器类型
LED
功耗
重量
支持制式
软件要求
```

不得使用一个通用板卡替代不同业务板、主控板和上联板。

## 6.5 第五步：建立几何模型

几何建模顺序：

1. 机框轮廓；
2. 安装耳、把手、风道、边框；
3. 槽位；
4. 板卡和模块；
5. 端口与连接器；
6. LED 与标签；
7. 光模块和线缆；
8. 材质、颜色和丝印。

优先保证尺寸、位置、数量和接口形态准确，再处理材质和光影。

## 6.6 第六步：建立实际资产实例

实际资产实例应记录：

- 设备名称；
- 序列号；
- 站点和机架位置；
- 硬件版本；
- 软件版本；
- 实际板卡配置；
- 实际端口状态；
- 设备生命周期状态；
- 维护责任人。

实际配置必须与真实设备清单或采集结果一致。

## 6.7 第七步：建立物理连接关系

必须建立：

- 电源 A/B 路；
- 接地；
- 主控主备；
- 板卡与背板；
- PON 端口与 ODF；
- 上联端口与汇聚设备；
- 风扇与风道；
- 光模块与光纤。

连接关系必须是可查询的数据对象，不得只画在线路图中。

## 6.8 第八步：建立逻辑业务关系

至少建立以下映射：

- 物理端口 ↔ 逻辑端口；
- PON 端口 ↔ ONU；
- ONU ↔ T-CONT / GEM Port；
- Service Port ↔ VLAN；
- VLAN ↔ 上联端口；
- ONU ↔ 客户或业务标识；
- 告警 ↔ 影响业务。

## 6.9 第九步：接入运行数据

运行数据来源可包括：

- SNMP / MIB；
- CLI；
- NETCONF / YANG；
- REST API；
- Syslog；
- TR-069 / TR-369；
- 厂家 EMS/NMS；
- Oneasy 资源与告警接口。

采集项至少包括：

| 类别 | 关键数据 |
|---|---|
| 设备 | 在线状态、CPU、内存、运行时长 |
| 板卡 | 在位、运行、温度、功耗、版本 |
| 风扇 | 状态、转速、档位 |
| 电源 | 输入电压、电流、A/B 路状态 |
| PON | 管理状态、运行状态、ONU 数、流量 |
| ONU | 在线、离线原因、光功率、距离、温度 |
| 上联 | 带宽、流量、错误包、丢包 |
| 告警 | 级别、对象、时间、恢复状态 |

## 6.10 第十步：建立历史状态

运行孪生不能只有当前值。

至少应支持：

- 24 小时趋势；
- 7 天趋势；
- 30 天趋势；
- 关键告警事件回放；
- 板卡更换前后对比；
- 光功率劣化趋势；
- ONU 在线率变化；
- 容量增长趋势。

历史数据应区分：

- 原始值；
- 聚合值；
- 最大值；
- 最小值；
- 平均值；
- 事件点。

## 6.11 第十一步：建立诊断模型

故障诊断至少采用以下链路：

```text
异常对象
→ 告警
→ 性能证据
→ 同类对象对比
→ 上下游关系
→ 可能原因
→ 影响范围
→ 建议动作
```

例如 ONU 批量离线：

```text
PON 端口异常
→ 同一端口多个 ONU 同时离线
→ 检查端口告警和光功率
→ 关联 ODF / Splitter / 光缆
→ 判断公共故障或单点故障
→ 计算受影响客户数
→ 给出排查顺序
```

诊断结果应包含：

- 结论；
- 证据；
- 置信度；
- 未确认项；
- 建议动作；
- 风险提示；
- 是否需要人工确认。

## 6.12 第十二步：测试和验收

测试至少覆盖：

- 几何准确性；
- 槽位与板卡规则；
- 端口数量；
- 资源 ID；
- 数据刷新；
- 告警关联；
- 拓扑关系；
- 历史回放；
- 权限控制；
- 浏览器性能；
- 异常数据处理。

## 7. 几何与视觉标准

### 7.1 尺寸与比例

- 设备外形应按实际毫米尺寸建模；
- 槽位宽度和间距应按机械结构比例；
- 端口数量、排布和连接器类型必须正确；
- 机架 U 位应按标准机架尺寸处理；
- 不得为了美观任意改变设备比例。

### 7.2 显示层级

应采用分级显示：

```text
远景：机房、机架、设备位置
中景：机框、板卡、模块
近景：端口、LED、标签
特写：端口状态和连接关系
```

### 7.3 视图要求

至少提供：

- 正面正投影；
- 三维透视；
- 侧视图；
- 顶视图；
- 爆炸视图；
- 业务拓扑图。

工程操作不得只依赖三维视图。精确端口识别应优先使用正面工程图。

### 7.4 状态颜色

建议状态编码：

| 状态 | 颜色 |
|---|---|
| 正常 | 绿色 |
| 提醒 | 黄色 |
| 重大告警 | 橙色 |
| 严重故障 | 红色 |
| 离线/未知 | 灰色 |
| 维护 | 蓝色 |

颜色必须配合文字、图标和告警代码，不能只依靠颜色表达。

## 8. 物理与逻辑关系标准

### 8.1 物理关系

必须支持从设备追踪到终端：

```text
Rack
→ Chassis
→ Slot
→ Board
→ PON Port
→ ODF
→ Cable
→ Splitter
→ ONU
```

### 8.2 逻辑关系

必须支持从 ONU 追踪到业务：

```text
ONU
→ T-CONT
→ GEM Port
→ Service Port
→ VLAN
→ Uplink
→ BNG / BRAS
```

### 8.3 双向查询

必须支持：

- 从端口查 ONU；
- 从 ONU 查端口；
- 从告警查影响客户；
- 从客户查设备路径；
- 从上联查承载的 PON 和业务；
- 从板卡查所有端口和告警。

## 9. 运行数字孪生要求

达到 L3 时，系统至少应具备以下工作模式：

1. **资产模式**：设备、板卡、序列号、版本、位置；
2. **健康模式**：综合健康度、告警、在线状态；
3. **温度模式**：板卡和风扇热力图；
4. **功耗模式**：整机、板卡、电源 A/B 路；
5. **容量模式**：PON、ONU、上联和资源利用率；
6. **告警模式**：告警对象和影响范围；
7. **历史模式**：状态趋势和事件回放；
8. **诊断模式**：原因、证据、置信度和建议。

## 10. 数据质量标准

### 10.1 数据状态

所有运行数据应标记状态：

| 状态 | 含义 |
|---|---|
| Real-time | 实时采集 |
| Recent | 最近一次有效值 |
| Stale | 已过期 |
| Simulated | 模拟数据 |
| Inferred | 推断数据 |
| Unknown | 未知 |

模拟数据不得在界面中伪装成真实设备数据。

### 10.2 时间标准

每条遥测和告警至少包含：

- 采集时间；
- 设备时间；
- 平台接收时间；
- 时区；
- 数据有效期。

### 10.3 数据冲突处理

当 SNMP、CLI、EMS 和数据库数据冲突时，应：

1. 保留原始值；
2. 标记数据来源；
3. 按可信优先级选择展示值；
4. 生成数据一致性告警；
5. 不得静默覆盖。

## 11. 故障诊断标准

### 11.1 诊断输出格式

```text
症状
对象
时间
证据
可能原因
置信度
影响范围
建议动作
验证方法
风险与边界
```

### 11.2 常见场景

应优先覆盖：

- 板卡离线；
- PON 端口 LOS；
- ONU 批量离线；
- ONU 频繁掉线；
- 光功率劣化；
- 上联拥塞；
- 风扇异常；
- 板卡高温；
- 电源单路故障；
- VLAN / 业务配置错误；
- ONU 在线但业务不可用。

### 11.3 人工确认

以下操作不得由数字孪生自动执行，除非有明确授权、审批和回滚机制：

- 板卡复位；
- PON 端口关闭；
- 批量 ONU 配置；
- VLAN 删除；
- 固件升级；
- 主备切换；
- 批量重启；
- 生产网络脚本执行。

## 12. 交互与界面标准

推荐布局：

```text
左侧：资产树和过滤器
中间：3D / 正面图 / 拓扑 / 历史视图
右侧：属性、状态、告警、性能、业务关系
底部：时间轴、事件、诊断过程
```

基本交互：

- 悬停：显示对象摘要；
- 单击：选中对象；
- 双击：进入对象详情；
- 右键：显示允许的运维动作；
- 告警点击：跳转并高亮影响路径；
- 缩放：按层级改变信息密度；
- 搜索：支持设备、槽位、端口、ONU、客户和告警。

## 13. 性能要求

### 13.1 前端性能

- 首屏应优先加载机框和主要板卡；
- 端口和 ONU 应按需加载；
- 使用 LOD 降低远景模型复杂度；
- 大量端口应采用实例化渲染；
- 不应一次加载全部 ONU 三维对象；
- 历史趋势和拓扑数据应分页或分区加载。

### 13.2 数据刷新

建议刷新策略：

| 数据 | 建议周期 |
|---|---:|
| 告警 | 事件推送或 5–15 秒 |
| 设备状态 | 15–60 秒 |
| 温度/功耗 | 30–300 秒 |
| 光功率 | 5–15 分钟 |
| 容量与报表 | 15–60 分钟 |
| 静态资产 | 变更触发 |

实际周期应根据设备能力、网络规模和系统资源确定。

## 14. 安全与权限标准

数字孪生系统涉及生产网络，应遵守：

- 最小权限；
- 只读与可操作权限分离；
- 高风险操作二次确认；
- 操作审批；
- 全量审计日志；
- 凭据不在前端暴露；
- SNMP、CLI、API 凭据加密；
- 多租户隔离；
- 数据脱敏；
- 生产与演示环境分离。

AI 诊断输出只能作为辅助建议，不能替代工程师对生产变更的最终确认。

## 15. 验收标准

### 15.1 L2 工程结构模型验收

必须满足：

- 设备尺寸与比例基本正确；
- 槽位编号完整；
- 板卡类型与槽位规则正确；
- 端口数量和位置正确；
- 物理对象均有唯一 ID；
- 正面、三维和爆炸视图可用；
- 模型未把未知信息冒充为事实。

### 15.2 L3 运行孪生验收

除 L2 外，还应满足：

- 可接入真实或明确标识的模拟数据；
- 设备、板卡、端口状态可更新；
- 支持温度、功耗、容量、告警视图；
- 支持历史趋势；
- 支持告警定位；
- 数据来源和时间可追踪；
- 数据过期和异常可识别。

### 15.3 L4 运维决策孪生验收

除 L3 外，还应满足：

- 物理和逻辑关系可双向查询；
- 故障影响可追踪至 ONU、客户或业务；
- 可输出结构化诊断；
- 诊断有证据和置信度；
- 支持工单、变更和维护记录；
- 高风险操作受审批和审计控制；
- 支持处理后验证和复盘。

## 17. OnExpert 建模工作规则

OnExpert 在执行接入网设备数字孪生任务时，必须遵守：

1. 先核对型号，再开始建模；
2. 优先使用厂家官方资料；
3. 先做结构正确，再做视觉优化；
4. 不得把不同型号或不同板卡混用；
5. 不得根据经验自行生成未证实端口；
6. 必须区分真实、模拟和推断数据；
7. 必须同时考虑物理路径和业务路径；
8. 必须设计故障影响分析能力；
9. 必须为后续 SNMP、CLI、Syslog 和 Oneasy 接口预留；
10. 每个版本完成后必须执行结构、数据、交互和诊断自检；
11. 用户要求“工程级”时，不得仅输出概念图或视觉效果图；
12. 用户要求“数字孪生”时，必须说明当前达到 L0–L4 中的哪个等级。

## 19. 总结

接入网设备工程级数字孪生的本质，不是把设备“画成立体的”，而是建立一套能够连接：

```text
设备结构
+ 网络资源
+ 运行数据
+ 业务关系
+ 运维过程
```

的统一工程模型。

最终标准可以概括为：

> **看得像、找得到、对得上、查得清、能诊断、可追溯。**

达到这一标准后，数字孪生才能真正成为 OnExpert 和 Oneasy 的重要技术资产，并用于设备认知、资产管理、运维监控、故障诊断、容量规划、工程培训和智能运维。
